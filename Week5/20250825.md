- `백승진`
    - **GitHub 프로젝트 정리 및 이슈 전략 정리**
        - 이슈들을 효과적으로 관리하기 위해 `Projects`에 **Module**이라는 커스텀 필드를 추가했습니다. 이를 통해 각 이슈가 어느 모듈 레이에 속하는지 파악하기가 훨씬 수월해졌습니다.

        - 프로젝트의 기본 모듈 레이어별로 **마일스톤**을 설정하고, 각 모듈별로 **Root Issue**를 만든 뒤 그 아래에 세부적인 **Sub-Issue**들을 정리했습니다.
    - **프로젝트 구현**
        - 기존의 **DailyTemperature** 테이블이 단기 예보에만 사용된다는 것을 알게 되어, 이 테이블을 **ShortForecastDetail** 테이블에 통합하기로 결정했습니다.
        - **DailyTemperature** 같은 자주 쓰는 값들을 **도메인 객체**로 만들고, **Forecast** 도메인 객체에 필요한 필드를 추가했습니다.
        - **Common** 모듈에 온도, 습도 등을 활용한 등산 활동 지수 계산 유틸 클래스를 만들 수 있었습니다. 이 유틸 클래스는 **Forecast** 도메인 객체에서 필드를 기반으로 등산 활동 지수를 반환할 때 사용하도록 하였습니다.

# ☀️ 어려웠던 점

- `김명규`
    - CD의 흐름에 대해서 이해하는 것과 각 서비스에 접근할 수 있는 권한 정책을 가진 Role을 설정하는 것이 매끄럽게 이어지지 않아서 오전 시간에 Role 관련 정책 설정에 대해서 고민하는 시간이 많았다.
    - Mountain 도메인 관련 코드를 작성해야 하는데 순수 자바로 구성된 `도메인` 과 Spring JPA에서 관리하는 `엔티티` 를 분리할 때 어떻게 하면 좋을지 막막하게 느껴졌다.
- `백승진`
    - 기존에 GitHub Projects를 사용해 본 경험이 없어 초반에 막막했습니다. 하지만 정해진 답이 있는 것이 아니니, 규칙을 정하고 진행하면서 필요에 따라 수정하며 진행해야겠다고 생각했습니다.
    - Common 모듈에 유틸 클래스를 작성하면서 과연 이 모듈이 꼭 필요한가에 대한 고민을 했습니다. 해당 계산 로직이 결국 예보 도메인에 종속적인 것 같다는 생각이 들었기 때문입니다.
- `박종성`
    - CI/CD, 도메인 설정을 처음하다보니 공부할 것들이 많았다. 
    서비스 하나하나의 역할들을 차분히 정리해볼 필요성을 느꼈다.
    - atoms 단위가 어려웠다. 리서치를 진행해도 추상적인 표현들이 많이 나와서 더 헷갈렸다.
    하지만 폴더 구조 등의 예시들을 많이 찾아보며 보성님과 의견을 좁혀나갔다.
- `시보성`
    - 아토믹 디자인 단위 기준 설정의 주관성
        - molecule과 organism의 경계가 모호해 팀원 간 판단이 다르게 나오는 경우가 많았음.
        - 명확한 설계 기준 없이 컴포넌트를 분리하다 보니 코드 리뷰 시 구조적 개선 방향을 잡기 어려웠음.
    - bottom-up 방식의 설계 난이도
        - 아토믹 디자인 패턴에 익숙하지 않아, 작은 단위에서 시작해 상위 컴포넌트를 구성하는 접근에 시간이 많이 소요되었다.
    - 중복 및 불필요한 Props 발생
        - 유연성을 확보하려다 보니 props가 과도하게 증가하는 문제 발생되어 atoms가 가지고 있는 디자인이 존재하지 않는 느낌이라 제대로 사용하지 못하고 있다는 생각이 강하게 들었다.
    - 비슷한 atoms가 생기는 경우 props로 추가할지 다른 방법을 사용할지 어떤 것이 깔끔할지 고민
        - compound 컴포넌트
        - Comment 예시를 통해 compound 패턴을 적용해 UI 변형에 유연하게 대응하는 방법 정리.

# ☁️ 개선할 점

- `김명규`
    - ERD를 보고 JPA entity 코드를 작성하는 과정에서 누락된 필드가 있었다. 미리 알아채서 다행이지 큰일 날 뻔 했다. 또 ERD를 작성할 때 나름 고심해서 짰다고 생각했는데 오늘 JPA entity를 작성하는 과정에서 오류를 발견할 수 있었다. 수정을 절대 안 해도 될 정도로 완벽하게 해내기는 힘들겠지만 그래도 조금 더 꼼꼼하게 작업하면 이런 일을 줄일 수 있을 것 같다.
    - FE, BE의 이슈, PR 제목 양식을 통일했고, 승진님이 레포 안의 projects를 잘 활용할 수 있도록 작업해주셨다. 이슈를 생성할 때 마일스톤, 상하위 이슈, 이슈 단위 등을 고려해서 효과를 200% 누려보자.
- `백승진`
    - 초기에 ERD를 더 꼼꼼히 설계했더라면 테이블 수정이 발생하지 않았을 텐데 하는 아쉬움이 남습니다. 계속해서 테이블이 수정되고 있는 걸 보면서, 처음부터 잘 설계했으면 이렇게 변경이 되지 않았을까 하는 생각도 들었습니다.
        
        하지만 이렇게 계속해서 구현을 해나가고 고민을 해가면서 **도메인 지식이 쌓이고, 어떻게 구현해야 할지 좀 더 명확해지는 느낌**을 받았습니다. 초기 설계가 완벽하지 않았더라도, 직접 코드를 작성하며 발생하는 문제들을 해결하는 과정에서도 배우는 것이 많다고 생각하기 때문에 이 경험이 다음 프로젝트의 설계에 큰 도움이 될 것 같습니다.
        
    - 강수량 데이터를 숫자 값으로 줄 거라고 예상했지만, "30mm 이상"과 같은 문구로 제공되는 경우도 있다는 것을 뒤늦게 알게 되어 당황했습니다. 사용자에게 보여주기 위한 데이터는 문제가 없지만, 등산 지수나 추천 문구 계산을 위해서는 숫자 값이 필수적이어서 곤란함을 느꼈습니다.
        
        API 키를 발급받아 확인했을 때는 이런 값이 없어서 배제했었는데, 문서를 더 꼼꼼하게 읽었어야 했다는 아쉬움이 남습니다. 앞으로는 API 명세서나 제공되는 문서를 꼼꼼하게 정독하는 습관을 들이겠습니다. 특히, 데이터 타입이나 예시 값을 놓치지 않도록 주의해야겠습니다.
        
- `박종성`
    - 마음이 급하다보니 차분히 하나씩 처리하는 것이 어려웠다. 
    급할수록 차분히 하나하나 생각하고 구현하는 것이 오히려 시간을 단축하고 꼼꼼히 하는 것임을 다시 한 번 새겨야할 것 같다.
- `시보성`
    - atomic을 덜 이해한 상태로 module.scss를 선택해서 프로젝트를 진행했던것이 아쉬운 점으로 남는다. 먼저 충분한 리서치를 하고 의견을 많이 나눈 상태로 들어갔더라면 초기 선택이 달라서 좀더 결과물을 다듬을 시간을 남길 수 있었을 것 같은데 빠른 의사결정으로 인해 많은 작업은 아니지만 다시 작업을 해야하는 상황이 생긴 것같다.
    - wiki를 처음 꾸며봤는데 이러한 기록을 남기는 것이 얼마나 중요한 것 인지 점점 실감을 하고 있는 것 같다. 이전에 프로젝트도 전부 이러한 문서를 남겨뒀더라면 더 기억에 잘남고 프로젝트에 짠 코드들을 더 발전시킬 수 있는 방향이 될 수 있지 않았을까 생각되어 이러한 점을 개선하고 싶다.
